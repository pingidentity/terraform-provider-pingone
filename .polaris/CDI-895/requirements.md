# Requirements

## Summary
The `pingone_application` resource enforces a Terraform-side maximum of 20 entries for `oidc_options.cors_settings.origins`, but the PingOne API allows up to 40. The fix requires changing the `originsMax` constant in `resource_application.go` from 20 to 40, and separately updating a hardcoded "Limited to 20 values." string in `data_source_application.go` (which does not reference the constant). The schema upgrade file and utils file do not contain any validator or limit that needs changing, and no tests assert the old limit of 20.

## Functional Requirements
1. Raise the `originsMax` constant from `20` to `40` in `internal/service/sso/resource_application.go` (line 1981). — Source: Jira CDI-895 AC / discuss Locked Decision 1
2. Update the hardcoded description string in `data_source_application.go` (line 776) from "Limited to 20 values." to "Limited to 40 values." — Source: Research finding: `data_source_application.go:776` uses a hardcoded string, not the `originsMax` constant.

## Non-Functional Requirements
1. The change is limited to the validator value and the description strings only — no API behavior, no data model, and no other attributes are affected. — Source: discuss Constraint 1
2. The `docs/resources/application.md` and `docs/data-sources/application.md` generated documentation must reflect the updated limit after running `make generate`. These files are auto-generated by `tfplugindocs` from Go source and do not need manual edits. — Source: Research finding: `GNUmakefile:32` runs `go tool tfplugindocs generate`; `docs/resources/application.md:285` contains `<!-- schema generated by tfplugindocs -->`.

## Technical Context

### resource_application.go (the authoritative source)
- `internal/service/sso/resource_application.go:1981` — `const originsMax = 20` is a function-local constant inside `resourceApplicationSchemaCorsSettings()`.
- `resource_application.go:1983` — The description string uses `fmt.Sprintf` with `%d` and the `originsMax` constant, so it will update automatically when the constant changes.
- `resource_application.go:2010` — `setvalidator.SizeAtMost(originsMax)` references the same constant, so the validator also updates automatically.

### data_source_application.go (requires a separate change)
- `internal/service/sso/data_source_application.go:776` — The origins description is a hardcoded string literal: `"...Limited to 20 values."`. It does NOT use the `originsMax` constant and will not update when `resource_application.go` changes. This string must be updated manually to "Limited to 40 values."
- The data source `origins` attribute is `Computed: true` with no `SizeAtMost` validator (data sources read-only), so no validator change is needed here — only the description string.

### resource_application_schema_upgrade_0_to_1.go (no change needed)
- `internal/service/sso/resource_application_schema_upgrade_0_to_1.go:705-709` — The V0 schema's `origins` attribute has no validator (`Optional: true` only). This is intentional for a state upgrade schema representing the old state shape. No numeric limit appears and no change is needed.

### utils_application.go (no change needed)
- `internal/service/sso/utils_application.go:74` — Only maps `GetOriginsOk()` to a Terraform string set. No numeric limit or validator. No change needed.

### Test files (no changes needed)
- `internal/service/sso/resource_application_test.go` — All CORS test assertions check counts of 8 origins or 0 origins. No test asserts a count of 20, checks the exact validator limit, or would break when the limit is raised.
- `internal/service/sso/data_source_application_test.go` — No assertions on the 20-origin limit.
- Beta helper test files — No assertions on the 20-origin limit.

### Generated documentation (regenerate, do not manually edit)
- `docs/resources/application.md` — Contains "Limited to 20 values." in three places; auto-generated from `resource_application.go`. Will update on `make generate`.
- `docs/data-sources/application.md` — Contains "Limited to 20 values." in two places; auto-generated from `data_source_application.go`. Will update on `make generate` after the description string fix.

## Constraints
1. Change is limited to the validator value and description strings only — no API behavior, no data model, no other attributes. — Source: discuss Constraint 1
2. The description string in `resource_application.go` is templated via `originsMax`, so it updates automatically — no separate string change needed there. — Source: discuss Constraint 2
3. The schema upgrade file (`resource_application_schema_upgrade_0_to_1.go`) represents the V0 (old) schema shape for state migration purposes and intentionally has no validator. It must not be changed. — Source: Research finding.
4. `docs/` files must not be manually edited; they are regenerated by `make generate`. — Source: Research finding: `GNUmakefile:32`.

## Acceptance Criteria
1. `const originsMax` in `internal/service/sso/resource_application.go` is set to `40`.
2. The `setvalidator.SizeAtMost(originsMax)` validator on `oidc_options.cors_settings.origins` enforces a maximum of 40, verified by plan/apply accepting a configuration with 21-40 origins that previously would have failed.
3. The description string for `origins` in `resource_application.go` reads "Limited to 40 values." (this follows automatically from the constant change).
4. The hardcoded description string in `internal/service/sso/data_source_application.go` (line 776) is updated from "Limited to 20 values." to "Limited to 40 values."
5. `resource_application_schema_upgrade_0_to_1.go`, `utils_application.go`, and all test files are unchanged.
6. `make generate` is run and `docs/resources/application.md` and `docs/data-sources/application.md` reflect "Limited to 40 values." in all relevant places.
7. All existing tests pass without modification.
